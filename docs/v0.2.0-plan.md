# twt v0.2.0 — Detailed Plan

## Overview

Close the feature gap with xurl. Add 20 new commands across 4 tiers, bringing twt from read-only to full read/write parity — still free, still using the private GraphQL API.

## Current State (v0.1.0)

10 commands implemented: `auth login/status/logout`, `user`, `tweets`, `tweet`, `follow/unfollow`, `like/unlike`, `search`, `timeline`.

API layer: `graphqlGet()`, `graphqlPost()`, `parseTimelineInstructions()`, `parseUserResult()`, `parseTweetResult()`. Rate limiting with jitter + 429 retry. Chrome cookie extraction.

## New Commands

### Tier 1 — Posting (highest value)

#### `twt post <text>`
- **Mutation**: `CreatePost` → `vMia9QJ2JVkCXuO5J4MTbw`
- **Variables**: `{"post_text": "<text>"}` — APK uses `post_text`, NOT `tweet_text`
- **Features**: `{"subscriptions_feature_can_gift_premium": true}`
- **Response path**: `data.create_tweet.tweet_results.result`
- **Output**: Print the created tweet URL + ID
- **Safety**: Confirmation prompt before posting. Skip with `--yes` flag.

#### `twt reply <tweet-id> <text>`
- Same `CreatePost` mutation
- **Variables**: `{"post_text": "<text>", "reply": {"in_reply_to_tweet_id": "<tweet-id>", "exclude_reply_user_ids": []}}`
- **Output**: Print reply URL

#### `twt quote <tweet-id> <text>`
- Same `CreatePost` mutation
- **Variables**: `{"post_text": "<text>", "attachment_url": "https://x.com/i/status/<tweet-id>"}`
- **Output**: Print quote tweet URL

#### `twt delete <tweet-id>`
- **Mutation**: `DeletePostMutation` → `1EVIme6zMCgTO7F95wuElA`
- **Variables**: `{"tweet_id": "<tweet-id>"}`
- **Safety**: Confirmation prompt. Skip with `--yes`.
- **Output**: `✓ Deleted tweet <id>`

#### `twt repost <tweet-id>` / `twt unrepost <tweet-id>`
- **Mutations**: `CreateRepostMutation` → `ydMACa-dOjZx126SWo6q5A` / `DeleteRepostMutation` → `w1Bo2Whh4f4lha5Djgnpvg`
- **Variables**: `{"tweet_id": "<tweet-id>"}`
- **Output**: `✓ Reposted tweet <id>` / `✓ Unreposted tweet <id>`

### Tier 2 — Bookmarks & Relationships

#### `twt bookmarks`
- **Query**: `BookmarksTimelineQuery` → `DN0j17CihaEo7QYmaZGkiw`
- **Variables**: `{"count": <limit>}`
- **Response path**: `data.bookmarks_timeline.timeline.instructions` → reuses `parseTimelineInstructions()`
- **Features**: timeline feature set

#### `twt bookmark <tweet-id>` / `twt unbookmark <tweet-id>`
- **Mutations**: `BookmarkAddMutation` → `IjefskW4Kr2i-6XRdshQEg` / `BookmarkRemoveMutation` → `K5KIqVnds5iJ00WdHb8Nmw`
- **Variables**: `{"tweet_id": "<tweet-id>"}`
- **Output**: `✓ Bookmarked tweet <id>` / `✓ Removed bookmark <id>`

#### `twt followers <handle>` / `twt following <handle>`
- **REST v1.1**: `/1.1/followers/list.json?screen_name=<handle>&count=<limit>` / `/1.1/friends/following/list.json?screen_name=<handle>&count=<limit>`
- These are REST, not GraphQL — need new `restGet()` method in `client.go`
- `restGet(path string, params url.Values) (json.RawMessage, error)` — same auth headers, same jitter, same rate limit handling
- **Response**: `{"users": [...]}` — each user has `screen_name`, `name`, `followers_count`, `description`
- **Note**: REST v1.1 users use the `legacy` schema (NOT APK schema). Need separate parsing or map to `types.User`.
- **Output**: List of users with handle, name, follower count

### Tier 3 — Feeds & Moderation

#### `twt mentions`
- **Query**: `NotificationTimelineQuery` → `jsOzc8RhpUpH5InTskP6Yw`
- **Variables**: `{"count": <limit>}`
- **Response**: Timeline format with notification entries — filter for mention-type
- **Note**: APK response path needs discovery (likely `data.notification_timeline.timeline.instructions` or similar). Test first.

#### `twt likes [handle]`
- **Query**: `UserProfileFavoritesTimelineQuery` → `M34xxhtrHWGAxGofYSGclA`
- **Variables**: `{"rest_id": "<user_id>", "count": <limit>}`
- Default: your own likes (use stored user ID from auth). With handle: resolve → ID first.
- **Response**: Timeline format, reuses `parseTimelineInstructions()`

#### `twt block <handle>` / `twt unblock <handle>`
- **Mutations**: `BlockUser` → `8zl3cVULtte29uCoWREtBQ` / `UnblockUser` → `WtUZ-1fkiAJGXfN6gAwrLw`
- **Variables**: `{"target_user_id": "<user_id>"}` — same pattern as follow/unfollow
- Resolve handle → ID via existing `GetUserByScreenName()`
- **Output**: `✓ Blocked @handle` / `✓ Unblocked @handle`

#### `twt mute <handle>` / `twt unmute <handle>`
- **Mutations**: `MuteUser` → `LoZAfbPr53jnw9Y2FydOIQ` / `UnmuteUser` → `29vlsCe7kkuB4JKnQGeK5w`
- **Variables**: `{"target_user_id": "<user_id>"}` — same pattern
- **Output**: `✓ Muted @handle` / `✓ Unmuted @handle`

#### `twt whoami`
- No new API call needed
- Read stored credentials → extract user info (either from saved handle in credentials.json, or call `GetUserByScreenName()` with the stored handle)
- **Prerequisite**: Store the user's handle/ID during `twt auth login` (currently we try via verify_credentials but it often returns empty). Fix: after cookie extraction, call `GetUserByScreenName` using the `twid` cookie value (which contains the user ID) or add a `GetUserByID` call.
- **Output**: Same profile card as `twt user` but for yourself

### Tier 4 — DMs & Media (complex, separate v0.3.0 candidate)

#### `twt dm <handle> <text>` / `twt dms`
- **Send**: `SendMessageCreateMutation` → `TWRPP7gnKwV_R8-tE-Dd3Q`
- **List**: No clear GraphQL query for DM inbox in the extracted operations. May need REST `/1.1/dm/inbox_initial_state.json` or similar.
- **Complexity**: DM conversations have their own data model (conversation IDs, message IDs, participant resolution). Significantly more complex than tweet operations.
- **Recommendation**: Defer to v0.3.0

#### `twt post --media <file>`
- **Upload**: REST `/1.1/media/upload.json` (multipart form, chunked for video)
- **Attach**: Include `media_id` in CreatePost's `media` variable: `{"media": {"media_ids": ["<id>"], "tagged_users": []}}`
- **Complexity**: Media upload is multipart REST with chunked upload for video (INIT → APPEND → FINALIZE). Different from all other API calls.
- **Recommendation**: Implement image upload in v0.2.0 (simple single-request multipart). Video upload deferred to v0.3.0.

## Implementation Order

1. **Auth fix** — store user ID + handle during login
2. **Tier 1** — post/reply/quote, delete, repost/unrepost
3. **Tier 2** — bookmarks, bookmark/unbookmark, followers/following (add restGet)
4. **Tier 3** — likes, mentions, block/unblock, mute/unmute, whoami
5. **Wire all into main.go**
6. **Update docs** — spec.md, README.md, CHANGELOG.md
7. **Test every command live**
8. **Codex review**
9. **Tag v0.2.0 → release → tap update**

## Risk Areas

1. **CreatePost response path** — APK may use `data.create_tweet.tweet_results` or a different nesting. Must test live and adjust.
2. **REST v1.1 followers/following** — Different auth format? Cookies should work with REST too (they do for verify_credentials). Must test.
3. **Mentions timeline** — `NotificationTimelineQuery` may return all notification types, not just mentions. May need client-side filtering or a different query.
4. **Variable names** — Every mutation has APK-specific variable names that differ from web docs. Always verify against the decompiled Java source before implementing.
5. **Bookmarks timeline path** — Response nesting unknown. Must test live first.

## What's NOT in v0.2.0

- DMs (complex conversation model, defer to v0.3.0)
- Media upload for video (chunked upload, defer to v0.3.0)
- Streaming / real-time (fundamentally different architecture)
- Raw API access (`twt raw /2/endpoint`)
- Multi-app management (single Chrome profile only)
